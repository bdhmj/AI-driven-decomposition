## Column definitions for Part 4 table

- **Категория**: Which module/area this relates to
- **Расширение**: Short name for the enhancement
- **Описание**: What exactly this adds or changes
- **Влияние на проект**: Approximate effort in days, or qualitative (low/medium/high)
- **Фаза**: MVP (if essential for base quality) or Post-MVP
- **Рекомендация**: Professional opinion — one of:
  - "Рекомендую для MVP" — should be in base scope for quality reasons
  - "Опционально" — nice to have, client decides
  - "Рекомендую для Post-MVP" — worth doing but not blocking launch
  - "Критично для production" — must have before going live, even if post-MVP

## Important notes

- **No Technology Stack section** — tech stack is determined during architecture review, not in spec
- **Part 1 is narrative**, accessible to non-technical stakeholders
- **Part 2 is precise**, uses unambiguous language for contractual clarity
- **Part 3 is bounded** — only questions that materially affect estimate/architecture
- **Part 4 is exhaustive** — include every reasonable enhancement, no artificial limit
